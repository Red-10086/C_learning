## 一句话核心

**分治法：把复杂问题拆成小块，逐个击破，再拼回整体。**

- **“自顶向下”** 是拆开（看地图）：先看森林，再看树木。
- **“自底向上”** 是拼回（攀岩）：先搞定每棵树，再认全整片森林。

两者必须结对使用。只拆不拼，是走马观花；只拼不拆，是盲人摸象。

## 一、两套代码阅读方法（互补，非对立）

### 方法A：自底向上——先零件，后组装（适合短小代码）

**适用场景**：代码较短（50行以内）、数据结构是核心、你想复用其中的模块。

**四步走**：

| 步骤 | 目标 | 检查的问题 |
| :--- | :--- | :--- |
| **1. 看 “演员”** | 找到所有 `struct` 和 `typedef` | “程序里有哪些‘东西’？” |
| **2. 看 “动作”** | 找接收结构体指针的函数 | “这些函数对东西做了什么？” |
| **3. 看 “导演”** | 看 `main` 的流程骨架 | “先做什么？后做什么？” |
| **4. 抠细节** | 语法糖和边界条件 | “`->` 和 `.` 为什么这里能用？” |

**核心心法**：像认人一样，先认识谁是主角配角，再看他们干了什么，最后看导演怎么安排。

---

### 方法B：自顶向下——先流程，后细节（适合中大型代码）

**适用场景**：代码较长（50行以上）、业务逻辑是核心、你想快速理解整体流程。

**4 轮协议（每轮绝不越界）**：

| 轮次 | 目标 | 边界（绝不做什么） | 核心问题 |
| :--- | :--- | :--- | :--- |
| **第1轮：宏观流程** | 追踪 `main()` 执行路径 | 不深入任何函数体，不关心数据结构细节 | “程序在干什么？输入/输出是什么？” |
| **第2轮：接口与结构** | 画数据装箱图、函数契约 | 不看函数实现，不关心格式化细节 | “数据怎么组织？接口承诺了什么？” |
| **第3轮：设计决策** | 追问 “为什么这么写” | 不调试、不优化、不找边界异常 | “为什么用 `const`？为什么值嵌套？” |
| **第4轮：边界与防御** | 只标记 “地雷”，不排雷 | 不写修复代码，不引入新函数 | “哪里会崩？输入越界会怎样？” |

**核心心法**：像剥洋葱，一层只剥一层皮。绝不提前看内层。

## 二、两套方法的共同根基

无论用哪种方法，本质上都是在做同一件事：**先分解，再深入**。

| 方法 | 第一层 “分解” 在哪个维度 |
| :--- | :--- |
| **自底向上** | 先按**数据 vs 动作** 分解（结构体是数据，函数是动作） |
| **自顶向下** | 先按**层次**分解（宏观流程 → 数据结构 → 设计意图 → 边界） |

**你不需要二选一。** 面对一个陌生代码库，你可以先用自顶向下看大概，再用自底向上深潜某个你感兴趣的函数。

## 三、自己写代码的分层实践（你的多文件模块化）

你在写 “学生排序系统”(明解C_exercise12_4) 时无师自通的做法，完全暗合分层思想：

| 文件 | 分层角色 | 说明 |
| :--- | :--- | :--- |
| `my_struct_sort.h` | **接口层（契约）** | 定义结构体、枚举、函数原型。向所有使用者承诺能力和接口格式。 |
| `sort_Name.c` / `sort_Height.c` | **实现层（算法）** | 具体的排序逻辑，调用 `swap` 和比较函数，不关心数据从哪来。 |
| `swap_junior.c` | **原子操作层** | 最底层的 “交换两个结构体” 动作，被所有排序函数复用。 |
| `print_junior.c` | **表现层（输出）** | 只负责格式化显示，不关心数据结构怎么来的。 |
| `exercise12_4.c` | **应用层（剧本）** | `main` 所在，负责流程（输入→排序→输出），不关心具体排序怎么实现。 |

**核心心法**：每一层只为上一层服务，且只依赖下一层的**接口**，不依赖**实现**。

## 四、“一定程度”原则：何时停止下潜？

你不需要拆到晶体管，不需要拆到汇编，甚至不需要拆到 `printf` 的源码。你只需要拆到**离你当前问题最近的那一层**。

- **判断标准**：看“黑箱”。
  - **不拆**：它稳定、可靠、你不需要修改它。比如 `printf`，你知道它能打印，你知道它的格式串怎么用，你不需要知道它内部怎么操作显存。
  - **要拆**：你正在学习它、怀疑它、或者需要修改它。比如 `strcmp`，你因为不确定它的行为，所以亲手写了一个 `should_Swap_string_Is_Not_Ascending` 来替代它，并在过程中测绘了它的边界。
- **“一定程度”** = 你对一个黑箱的信任程度。信任不够，就拆开看看；信任够了，就留在外面当它是个可靠的积木块。
- **停止信号**：当你感觉“我懂了，它能稳定工作了”，就停手。当你感觉“这里唬我，我不确定”，就继续下潜。
- **进阶信号**：当你在更高层的工作中，遇到了一个因为底层黑箱行为导致的Bug（比如你之前二分查找的边界问题），那时候再回来拆那个特定黑箱的具体一层。

### 🧠 这不是“偷懒”，是“按需分配认知资源”

你之前在C语言里深潜 `const` 的裂缝、`strncpy` 的 `\0` 陷阱、指针的二象性，是因为那些是你当时正在测绘的“边界公理”。你完全没有去深潜 `printf` 的源码——因为它稳定、可靠、你不怀疑它。

这种“按需分配”，正是你“一定程度”原则的最好示范。你不是在“放弃深度”，你是在“把深度留给真正需要的地方”。

## 五、与已有公理系统的咬合

本笔记需与你此前建立的核心公理联动：

- **骨架词 vs 血肉词**：分层思想是骨架词（必须内化），4 轮阅读协议是血肉词（通过练习建立体感）。
- **边界意识**：每一层都有边界。阅读的边界是 “本轮绝不做什么”，编码的边界是 “头文件只暴露接口，不暴露实现”。
- **黑箱与白箱**：自底向上的深度，就是决定把一个东西当 “黑箱” 还是拆成 “白箱”。
